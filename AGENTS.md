## What Relay is

Relay is an **open-source, self-hosted replacement for AWS Systems Manager
Incident Manager** (which is end-of-life). It supplies only the orchestration
layer AWS abandoned — **on-call scheduling, escalation policies, dual-stream
incident routing, and a live fleet dashboard** — and delegates alarm detection,
SMS/email transport, ticketing, and runbooks to the AWS and enterprise services
that already do them well.

It runs as **one always-on container** (ECS Fargate, or any container host). A
CloudWatch alarm flows in via one EventBridge rule → SQS → the container, and the
whole life of the incident (classify → page on-call → escalate → turn the tile
red → file a ticket) happens in that one process, visible in one log stream. See
[`docs/architecture.md`](docs/architecture.md).

You deploy Relay into **your own AWS accounts**. It never phones home. Apache-2.0.

## Two topologies (there are exactly two)

| Topology | What it is |
|---|---|
| **team** (default) | One container + one DynamoDB table in a team's own account. Detection, paging, escalation, and the dashboard all run there. |
| **federated-hub** | The same image run as an org-wide aggregator / NOC big-board. Team deployments forward SEV1/SEV2 up to it. The hub stores **no static catalog** — it builds the org hierarchy from what teams push up via heartbeats. |

The same container artifact is both — its role is chosen by config
(`RELAY_DEPLOY_TYPE`, `RELAY_HUB_SCOPE`), not by a different build.

## The config / PII split — read before touching config

**No PII in Git, ever.** This is a hard rule, not a preference.

| Data | Where | Why |
|---|---|---|
| Escalation policies, routing rules | Git (`config/escalation.yaml`, `config/routing.yaml`) — startup **seed** | Reviewable, reproducible; reference opaque `contact_id` values only — never names/emails/phones |
| Routing + ignore **rules at runtime** | DynamoDB (UI-managed) — **DB wins at runtime**; the YAML block is a startup seed | Instant edits without redeploy |
| On-call schedules & availability | DynamoDB in your account | Generated by the auto-scheduler; changes weekly; never in Git |
| Contact PII (name, email, phone) | DynamoDB in your account, encrypted at rest | Never leaves the account; no PII in shared repos |
| Integration credentials (GitLab/ServiceNow/Teams) | DynamoDB settings table (encrypted), set on the **Settings** UI at runtime | No secret to pre-create; a Settings value overrides any deploy-time env fallback |

Escalation configs page a **role** (primary / secondary / manager); the schedule
resolves the role to the on-call person at page time. See
[`config/README.md`](config/README.md) and [`docs/configure.md`](docs/configure.md).

## Guardrails (design constraints you must not violate)

- **AI augments; it never gates.** The page fires immediately on `TRIGGERED`. All AI output is async, labeled AI-generated, attached after the fact. Never make paging wait on AI.
- **No git write on the hot path.** Config is read from memory; Git is written only on slow/human paths.
- **Durable escalation.** Timers are DynamoDB deadlines fired by a 30s sweep loop — they survive a container restart. Do not replace with an in-memory timer or external scheduler.
- **One EventBridge rule.** Every alarm flows through one rule — no per-alarm wiring.
- **Read-only skills.** Skill `probe.sh` scripts use only `describe*`/`list*`/`get*`/`lookup-events`/`filter-log-events` — never a mutating call.
- **Core stays AWS-free.** Keep concrete AWS/integration/AI behind adapter interfaces; don't import `boto3` into `core/`.
- **AWS-first, but modular.** Only AWS apps are monitored today; keep provider-specific code behind adapters so this stays extensible.
- **Two topologies only.** Don't reintroduce "standalone" / "central" as separate builds. The `Stream.CENTRAL` enum must **not** be renamed (it names a notification stream, not a topology).
- **Public-repo hygiene.** Never name specific customers or government agencies in committed docs/code — say "government agencies" generically. Don't disclose our own infra/security posture; environment-specific deploy config goes in gitignored `scripts/*.local.sh`.
- **No PII in Git** (restated because it matters most).

## Working conventions

- **Definition of Done** is enforced: `scripts/relay-verify.sh` + the checklist in [`CONTRIBUTING.md`](CONTRIBUTING.md). `mypy` is blocking. Run `/dod` (or the script) before calling work shippable.
- **Ship docs with the fix.** When you change behavior, update the relevant `docs/` page **and** `docs/status.md` in the *same* change, and cross-link.
- **Feature branch for enhancements** — don't commit directly to `main`.
- **Work tracking is GitHub issues**, not static markdown checklists. `docs/status.md` is the code-verified ledger, not a todo list.
- The mkdocs site builds with `--strict` in CI — a broken nav entry or link fails the build. If you add a doc page, add it to `mkdocs.yml` `nav`.

./scripts/relay-verify.sh

./scripts/relay-fire.sh
