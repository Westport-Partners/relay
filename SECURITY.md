# Security Policy

## Supported versions

Relay has not yet made a tagged release. Security fixes are applied to the `main`
branch. Once versioned releases begin, this section will list which versions receive
security patches.

## Reporting a vulnerability

**Please do not report security vulnerabilities through public GitHub issues.**

Instead, report them privately via GitHub's [private vulnerability
reporting](https://docs.github.com/en/code-security/security-advisories/guidance-on-reporting-and-writing-information-about-vulnerabilities/privately-reporting-a-security-vulnerability)
("Report a vulnerability" under the repository's **Security** tab), or contact Westport
Partners at https://www.westportpartners.com/.

Please include:
- A description of the issue and its impact.
- Steps to reproduce (proof-of-concept if possible).
- Affected version/commit.

We aim to acknowledge reports within a few business days and will keep you updated on
remediation.

## Scope

Relay is **self-hosted**: you deploy it into your own AWS account(s) and it does not
phone home. The most security-relevant areas:

- IAM permissions generated by the CDK stacks (`infra/`) and the BYOR inline policies.
- Handling of secrets (GitLab token if using the GitLab config source, ServiceNow credentials)
  — sourced from AWS Secrets Manager, never committed.
- Contact PII — stored in the team's own DynamoDB table, encrypted at rest; never in Git.
- The Hub dashboard endpoint (ALB) and its exposure.

## Good practices for operators

- Run the Hub behind TLS (provide an ACM certificate) and restrict dashboard access.
- Scope the deploy/runtime IAM to least privilege (see `infra/RUNNER_IAM.md` and
  `docs/byor.md`).
- Rotate any integration secrets (GitLab token if using the GitLab config source, ServiceNow
  credentials) per your organization's policy.
